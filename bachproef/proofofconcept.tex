\section{Doel en Scope}
De Proof of Concept (PoC) heeft als doel aan te tonen hoe een bestaande CI/CD-pipeline uitgebreid kan worden met CRA-relevante securitycontroles, zonder het ontwikkelproces volledig te hertekenen. De focus ligt op twee kernactiviteiten: het genereren van een Software Bill of Materials (SBOM) en het automatisch uitvoeren van dependency scanning met een duidelijke kwaliteitsdrempel (quality gate) op basis van CVSS-scores.

De PoC is bewust beperkt gehouden tot een paar projectrepositories met relatief eenvoudige buildketens (zoals Maven- of .NET-projecten). Complexere scenario's zoals multi-repo orkestratie of geavanceerde policy engines vallen hier buiten de scope van deze bachelorproef.

\section{Gebruikte tools en omgeving}

\subsection{Gekozen tools}

De implementatie van de Proof of Concept maakt gebruik van Jenkins als CI/CD-platform, aangezien dit reeds binnen Excentis wordt toegepast. Voor de generatie van SBOMs werd Syft gekozen, omdat deze tool open source is en op een gebruiksvriendelijke manier CycloneDX-SBOMs kan genereren. Voor vulnerability scanning werd OWASP Dependency-Check ingezet, aangezien deze tool toelaat om bekende kwetsbaarheden in gebruikte dependencies op een relatief eenvoudige manier te detecteren en te rapporteren.

\subsection{Opzet van de testomgeving}

De PoC is opgezet in een Virtuele Machine (VM) waar een Jenkins Docker container op draait, zodat de integratie van securitystappen zo realistisch mogelijk is. De volgende componenten worden gebruikt:
\begin{itemize}
    \item Een demo-applicatie met externe dependencies om realistische SBOMs en dependency-rapporten te kunnen genereren.
    \item Syft als SBOM-generator, geconfigureerd om een CycloneDX-SBOM (\texttt{sbom.cdx.json}) te produceren op basis van de gecompileerde applicatie.
    \item OWASP Dependency-Check als Software Composition Analysis (SCA)-tool, gebruikt om kwetsbaarheden in de gebruikte libraries te detecteren en een HTML-rapport te genereren.
\end{itemize}

Alle pipelineconfiguratie is opgeslagen in een \texttt{Jenkinsfile}, zodat wijzigingen in de beveiligingslogica versiebeheer doorlopen zoals gewone broncode.


\section{High-level pipeline-overzicht}

De pipeline volgt een klassieke CI/CD-volgorde waarin de securitystappen geïntegreerd worden als kwaliteitscontroles. Na codewijzigingen wordt de applicatie opgebouwd en getest. Vervolgens wordt een SBOM gegenereerd op basis van de gecompileerde applicatie. Daarna wordt een dependency scan uitgevoerd om bekende kwetsbaarheden in externe libraries te detecteren. Indien de scanresultaten boven de ingestelde drempel uitkomen, wordt de pipeline afgebroken en wordt een rapport gegenereerd voor de ontwikkelaars.

Deze opbouw zorgt ervoor dat securitycontroles niet losstaan van het ontwikkelproces, maar deel uitmaken van dezelfde geautomatiseerde workflow. Op die manier wordt de feedback voor ontwikkelaars vroeg in het proces gegeven en blijft de impact op de bestaande werkwijze beperkt.

\section{Implementatie van de pipeline}

De pipeline is geïmplementeerd als een \textbf{declaratieve Jenkins pipeline} in een \texttt{Jenkinsfile}. Deze pipeline volgt de klassieke CI/CD-volgorde met securitystappen geïntegreerd als kwaliteitscontroles na de reguliere buildfase.

\subsection{Pipeline stages}

De pipeline bestaat uit de volgende stages:

\begin{enumerate}
    \item \textbf{Checkout}: Afhalen van de broncode uit de Git repository.
    \item \textbf{Build}: Compilatie van de Java applicatie met Maven (\texttt{mvn clean package}).
    \item \textbf{SBOM-generatie}: Genereren van een CycloneDX SBOM met Syft:
    \begin{verbatim}
    syft packages target/demo-app.jar \
        -o cyclonedx-json=sbom.cdx.json
    \end{verbatim}
    \item \textbf{Dependency scanning}: Uitvoeren van SCA met OWASP Dependency-Check:
    \begin{verbatim}
    dependency-check \
        --project "Demo App" \
        --scan "target/" \
        --format HTML \
        --out "reports/"
    \end{verbatim}
    \item \textbf{Quality Gate}: Valideren of het aantal hoge en kritieke kwetsbaarheden (CVSS > 7.0) onder de drempel blijft.
    \item \textbf{Rapportage}: Archiveren van SBOM en rapporten, notificatie naar developers.
\end{enumerate}

\subsection{Configuratie van security tools}

\textbf{Syft configuratie} (gedeelte van Jenkinsfile):
\begin{verbatim}
stage('SBOM Generation') {
    steps {
        sh '''
            syft packages target/demo-app.jar \
                --exclude "platform:java" \
                -o cyclonedx-json=sbom.cdx.json
            archiveArtifacts 'sbom.cdx.json'
        '''
    }
}
\end{verbatim}

\textbf{OWASP Dependency-Check configuratie}:
\begin{verbatim}
stage('Dependency Scan') {
    steps {
        sh '''
            dependency-check \
                --project "Demo App" \
                --scan "target/" \
                --format HTML,JSON \
                --out "reports/" \
                --suppression "suppressions.xml"
        '''
        archiveArtifacts artifacts: 'reports/**', allowEmptyArchive: true
    }
}
\end{verbatim}

\subsection{Quality Gate implementatie}

De quality gate controleert of het aantal kwetsbaarheden met CVSS-score > 7.0 nul is:
\begin{verbatim}
stage('Quality Gate') {
    steps {
        script {
            def criticalVulns = sh(
                script: 'grep -c "CVSS: [7-9]" reports/dependency-check-report.json || true',
                returnStdout: true
            ).trim()
            
            if (criticalVulns.toInteger() > 0) {
                error "Kritieke kwetsbaarheden gedetecteerd: ${criticalVulns}"
            }
        }
    }
}
\end{verbatim}

Deze implementatie zorgt ervoor dat alleen code zonder hoge of kritieke kwetsbaarheden door de pipeline raakt, terwijl SBOMs automatisch worden gegenereerd en gearchiveerd voor compliance-doeleinden.

\section{Technische Uitdagingen}

\subsection{Tool-integratie}

De eerste technische uitdaging, en degene die ook het moeilijkste zal zijn om te implementeren, bij het opzetten van een DevSecOps-pipeline is de integratie van meerdere securitytools in één consistente workflow. Elke tool heeft een specifiek configuratieformaat, authenticatie- en autorisatiemechanismen, en verwacht andere input- en outputformaten (zoals SBOM in CycloneDX, scanraporten in JSON of HTML). Dit verhoogt de complexiteit van de pipelineconfiguratie en maakt foutafhandeling moeilijker wanneer één van de tools faalt.

Bovendien moet er gekozen worden op welk moment in de pipeline SBOM-generatie en kwetsbaarheidsscans worden uitgevoerd. Built-time integratie levert nauwkeurige SBOMs op die overeenkomen met het artefact dat gedeployed wordt, maar vraagt een goede inpassing in bestaande build-scripts (Maven, npm, Docker,...). Als deze tooling niet correct wordt aangeroepen, bestaat het risico op onvolledige SBOMs of ontbrekende scanresultaten, wat de betrouwbaarheid van de hele keten ondermijnt.

\subsection{Performance Overhead}

Het toevoegen van SBOM-generatie en securityscans introduceert onvermijdelijk extra runtime in de CI/CD-pipeline. SBOM-tools moeten alle dependencies analyseren terwijl SCA- en SAST-tools vaak volledige codebases of container-images doorlopen. Zeker bij grotere projecten kan dit minuten toevoegen aan elke build, wat ontwikkelaars ontmoedigt om regelmatig te mergen en pipelines uit te voeren. Het komt er op neer dat hoe meer code en hoe meer dependencies er zijn, hoe langer het duurt voordat de pipeline volledig gerunt is.

Daarnaast kan ook opslag en transport van data een probleem zijn. SBOMs genereren en verwerken voor meerdere services, is veel werk. Afhankelijk van hoe groot de applicatie is en hoeveel dependencies het heeft kan dit gigantisch veel opslag vereisen. Dit moet ook nog eens 10 jaar opgeslagen blijven per versie, dus er moet een oplossing zijn voor lange termijns opslag van de SBOMs \autocite{CRAEvidenceSBOM}.

\subsection{False positives}

Een derde technische uitdaging is het omgaan met false positives in kwetsbaarheidsscans. SCA-tools matchen componenten vaak op basis van naam en versie tegen publiek beschikbare CVE-databases. Bij libraries die bijna dezelfde naam hebben in verschillende ecosystemen of bij slecht onderhouden CPE-gegevens kunnen kwetsbaarheden ten onrechte aan een project worden gekoppeld. Dit resulteert in ruis in de rapporten en extra werk om issues manueel te verifiëren en op te lossen.

Er is ook een andere kant dan de technische impact bij false positives, de procesmatige impact. Als ontwikkelaars continu als "kritiek" gemarkeerde kwetsbaarheden moeten negeren, vermindert het vertrouwen in de tooling en worden waarschuwingen sneller genegeerd. Een robuuste DevSecOps-pipeline moet daarom mechanismen voorzien om onterechte meldingen te onderdrukken, baseline-vergelijkingen te maken en enkel nieuwe of bevestigde kwetsbaarheden als blocking te behandelen.

\subsection{CRA-specifieke gaps}

Hoewel algemene DevSecOps-praktijken goed gedocumenteerd zijn, blijven er specifieke gebrekken voor CRA-conformiteit.

\textbf{Geen standaard CRA-checklists voor pipelines.} Er bestaan wel algemene DevSecOps-checklists, maar geen officiële of gestandaardiseerde CRA-specifieke controlelijsten voor CI/CD-pipelines. Dit dwingt teams om zelf te interpreteren hoe de CRA (zoals SBOM-kwaliteit, vulnerability handling en conformiteitsbewijs) te vertalen naar concrete pipeline-stappen.

\textbf{Onvolwassen SBOM → CVE-automatisering.} Ondanks dat CRA SBOMs vereist om vulnerabilities te identificeren, blijft de koppeling tussen SBOMs en CVE-databases (NVD, EPSS,...) nog experimenteel. Tools zoals SBOM-CVE-Check bestaan, maar vereisen vaak manuele configuratie en missen robuuste integratie met commerciële pipelines. Dit maakt automatische compliance-validatie onbetrouwbaar.
